The deterministic layer

Most of a Rails AI toolchain is LLM-driven — coders, reviewers. These are the other half: fast, exact, reproducible, gate-able in CI. They also run as a grounding pre-pass for the LLM reviewers in dte-skills before its lenses fire.

This repo is a staging home. Checkers land here first; once a category can stand on its own it graduates into a focused plugin — the way the Hotwire checkers became hotwire-rails-toolkit.

Skills

Two shipped. Each script runs standalone against any Rails app and exits non-zero on findings.

layer-boundary-lint

Rails layering violations the architecture books describe but no tool gates:

  • Current. read in models
  • request/params in the domain layer (services and interactors)
  • raw .where/.order/.joins in controllers/views
  • mailer/HTTP I/O in after_* callbacks
  • off-layer Current writes · oversized Current
Ships lint_layer_boundaries.sh (bash + python3) · reference guide

rails-test-smell-checker

Test smells across Minitest and RSpec:

  • sleep in system tests
  • missing WebMock.disable_net_connect!
  • stubbing the system under test
  • has_css?/has_content? inside an assertion
  • caveated heuristics (Mystery Guest)
Ships lint_test_smells.sh (python3) · reference guide

Install

As standalone tools, or as a Claude Code plugin.

run the scripts directly
# exits non-zero on findings — CI-friendly
skills/layer-boundary-lint/scripts/lint_layer_boundaries.sh app/
skills/rails-test-smell-checker/scripts/lint_test_smells.sh app/
as a Claude Code plugin
claude plugin marketplace add davidteren/dte-rails-toolkit
claude plugin install dte-rails-toolkit@dte-rails-toolkit-marketplace

layer-boundary-lint: rules are individually skippable by label; STRICT=1 fails on advisories; CURRENT_ATTR_MAX tunes the threshold. rails-test-smell-checker: high-confidence smells fail; heuristics only fail under --strict.

The bar for adding a checker

Three rules. A checker ships only if it clears all three.

Deterministic, names its ceiling

Heuristic text/AST scans, not full parsers — each script says so. A clean run is a gate, not a proof.

Verified both ways

Run against a real app (must be truthful — no false positives) and a synthetic broken case (must flag it).

False positives are the failure mode

A rule that can't be made high-confidence cheaply is demoted to a caveated heuristic or dropped — never shipped loose.
